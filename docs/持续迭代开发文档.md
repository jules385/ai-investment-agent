# A股AI投研系统持续迭代开发文档

> 版本：v0.4.0
> 运行模式：Codex 本地开发 + MCP 数据工具 + Obsidian 知识库
> 文档用途：记录项目开发约定、已踩过的问题、验证清单和后续迭代注意事项。

## 一、项目定位

本项目是一套本地化 A 股投研工作流系统。Codex 负责研究执行、分类判断、代码开发和入库复核；MCP 负责获取财务、资金和技术指标；Obsidian 负责沉淀可迭代的个人投资知识库。

当前项目不再以网页前端部署为主线，也不再依赖 DeepSeek、Anthropic 或其他外部 LLM API 完成入库。这里的 AI 指当前执行项目的 Codex agent。

核心目标不是生成一次性研报，而是持续维护：

- 行业知识库
- 个股四面逻辑
- 投资决策记录
- 单股跟踪面板
- 每日投研信息雷达
- 原始研究来源索引

## 二、当前关键架构

### 1. 多智能体投研

四个领域分析师负责独立研究：

- 基本面：行业、财务、估值、增长逻辑。
- 技术面：趋势、指标、支撑阻力、交易锚点。
- 筹码面：股东、资金流、北向、融资、龙虎榜。
- 情绪面：舆情、市场温度、散户行为。

首席分析师负责综合判断和报告合成。数据 QA 用于交叉验证。

### 2. Obsidian 入库结构

研究成果入库后不再写成单个个股大文件，而是拆为：

```text
obsidian-vault/
├── 行业知识库/<行业名>.md
├── 个股逻辑/<代码-名称>/
│   ├── _index.md
│   ├── 基本面.md
│   ├── 技术面.md
│   ├── 筹码面.md
│   ├── 情绪面.md
│   ├── 投资决策.md
│   └── 原始报告/
└── 跟踪面板/<代码-名称>.md
```

设计原则：Obsidian 是当前知识库，不是按报告来源堆叠的研报仓库。
`原始报告/` 是原文归档区，入库时由 Python 代码把 `reports/` 中的 Markdown 文件一模一样复制进去。这一步不经过 Codex 判断，不进入 archive-plan，不做摘要、排版或内容清洗。

## 三、入库工作流原则

### 1. 初次覆盖和后续跟踪分流

初次覆盖用于建立基线：

- 四面逻辑完整进入对应文件。
- 投资决策进入 `投资决策.md`。
- 行业共性内容进入行业知识库。

后续周度、月度、季度、年度跟踪默认不进入四面正文：

- 没有重大逻辑变化：只在 `投资决策.md` 的“跟踪记录”写一行结论，并指向原文。
- 有重大逻辑变化：更新对应四面文件或行业知识库。
- 只是季度数据、技术指标、资金数据例行刷新：不打乱四面基线。

### 2. 禁止重复来源壳

已废弃的输出形式：

```text
## 原文摘录：xxx
> [!quote] 来源
> 分类理由：xxx
```

这些内容会污染阅读体验。来源信息统一进入：

- `obsidian-vault/原始资料索引/sources.md`
- `投资决策.md` 的“跟踪记录”

校验器已经把 `原文摘录：` 和 `[!quote] 来源` 作为问题检查。

### 3. archive-plan 字段

`tools/archive-to-vault.py --emit-plan` 会生成知识入库计划。Codex 只负责行业知识库、四面逻辑和投资决策日志的分类判断，不负责 `原始报告/` 归档。原始报告归档由 Python 直接复制所有 Markdown 原文。

Codex 填写计划时重点关注：

```json
{
  "destination": "industry | stock | skip",
  "knowledge_type": "财务验证 | 技术趋势 | 筹码变化 | 情绪变化 | 投资决策 | ...",
  "target_file": "个股逻辑/<代码-名称>/基本面.md",
  "target_section": "目标章节",
  "update_mode": "create_baseline | append_update | replace_summary | source_only | skip",
  "impact_level": "major | minor | none",
  "logic_change": true,
  "update_policy": "merge_into_logic | decision_log_only | industry_update | source_only | skip",
  "effective_date": "YYYY-MM-DD",
  "action": "copy | skip",
  "reason": "分类理由"
}
```

默认策略：

- `01-初次覆盖/`：`merge_into_logic`
- `02-周度跟踪/`、`03-月度跟踪/`、`04-季度跟踪/`、`06-年度跟踪/`：`decision_log_only`
- QA、YAML、工具调用、流程记录、免责声明：`skip`

## 四、核心脚本职责

| 文件 | 职责 |
|------|------|
| `tools/archive-to-vault.py` | 入库主入口，导出计划、执行计划、校验 vault 文件 |
| `tools/archive_schema.py` | archive-plan 数据结构和字段校验 |
| `tools/archive_router.py` | 根据标题和来源推断目标文件 |
| `tools/archive_writer.py` | 创建基线文件、追加逻辑、写投资决策日志、保护跟踪面板 |
| `tools/archive_validator.py` | 检查 marker、乱码、废弃来源壳 |
| `tools/archive_formatter.py` | YAML frontmatter、AI 内容边界和基础排版 |
| `tools/archive_indexer.py` | 来源索引和入库统计结构 |

重要边界：

- `archive_writer.ensure_baseline_files(..., overwrite=True)` 不能覆盖已有 `跟踪面板/<代码-名称>.md`，跟踪面板是长期维护资产。
- `archive_writer.copy_raw_reports()` 只复制原始 Markdown 报告，不得修改正文内容，也不得依赖 AI 判断。
- `append_decision_log()` 用于后续跟踪，避免整篇报告进入四面正文。
- `build_block()` 不应恢复“原文摘录”包装。

## 五、重复出现的问题

### 1. Windows 中文路径和编码

PowerShell 在某些场景下写入中文 JSON 会把中文变成 `????`。之前生成 `archive-plan` 时出现过目标路径乱码。

处理原则：

- 涉及中文 JSON 写入，优先用 Python 或 Node 以 UTF-8 写入。
- 读写文件统一显式使用 `encoding="utf-8"`。
- 工具输出需设置 stdout UTF-8，`archive-to-vault.py` 已处理。

验证方式：

```bash
python tools/archive-to-vault.py --validate <vault文件>
```

并额外搜索：

```bash
rg -n "原文摘录：|\[!quote\] 来源|分类理由|\?\?\?" obsidian-vault
```

### 2. 后续跟踪过度堆叠

问题表现：

- 季度跟踪完整进入 `基本面.md`。
- 四面文件越来越长，核心逻辑被报告过程冲散。
- 同一来源、同一标题反复出现。

当前解决：

- 后续跟踪默认 `decision_log_only`。
- 只有 `logic_change=true` 且 `impact_level=major` 才更新四面或行业知识库。
- 数据更新要带日期，但不单开“来自季度报告”的栏目。

### 3. 单文件个股逻辑残留

旧结构曾生成：

```text
obsidian-vault/个股逻辑/<代码-名称>.md
```

新结构必须使用目录：

```text
obsidian-vault/个股逻辑/<代码-名称>/
```

如果同一个标的同时存在单文件和目录，要优先迁移并删除旧单文件，避免 Obsidian 中出现两套入口。

### 4. 跟踪面板被覆盖

`--overwrite` 重新入库时，曾把单股跟踪面板清空成模板。

当前规则：

- 跟踪面板只在不存在时创建。
- 已存在的跟踪假设、指标、日志和下次任务不得被覆盖。

### 5. 文档口径滞后

README、AGENTS、Skill 和命令文档容易出现旧口径，例如：

- 仍说入库需要外部 API。
- 仍说个股逻辑是单文件。
- 仍说后续跟踪完整搬运。
- 仍说 Stage 3 由外部 AI 排版。

开发时如果改了工作流，必须同步：

- `README.md`
- `AGENTS.md`
- `docs/使用手册.md`
- `skills/analysts/analyst-archive/SKILL.md`
- `commands/archive-research.md`

### 6. 命令入口乱码或过期

本项目的斜杠命令文件是用户实际触发工作流时最先接触的说明。曾出现 `commands/` 下多份文件被二次编码成乱码，导致 `/help`、`/analyze-initial`、`/archive-research` 等入口虽然文件存在，但用户和 Agent 读到的是不可用说明。

处理原则：

- 修改 Skill 或工具链后，必须同步检查 `commands/` 下对应入口。
- 命令文件应保持短、清晰、可执行，不要复制大段 Skill 内容。
- 首席分析师类命令必须明确：Chief 负责编排，禁止自己直接搜索数据或调用 MCP。
- 日报命令必须明确：日报是新闻整理，不默认调用行情、资金、估值、技术 MCP。

验证方式：

```bash
rg -n "浣犳槸|鎵ц|璺熻|鍛戒护|鐢ㄦ埛|鈥" commands
python tools/self-test.py
```

### 7. 自测脚本规则滞后

`tools/self-test.py` 是判断系统是否可用的入口之一。曾出现两个问题：

- 自测脚本本身中文乱码，输出不可读。
- 旧规则不了解新增的日报工作流，误报 `analyst-daily` 缺少数据源优先级或角色指令。
- 默认执行在线数据连通，导致网络受限时把外部源失败误判成系统本地工作流失败。

当前原则：

- 默认自测只检查本地结构：MCP 配置、Skill、Command、Python 语法、日报本地链路、核心 vault 样本。
- 外部行情/财务/新闻源连通只作为可选项，使用 `python tools/self-test.py --online`。
- 每新增一个命令、Skill、MCP 或核心工具，都要更新 `self-test.py` 的期望清单。
- 自测失败要区分“本地结构坏了”和“外部网络源暂时不可达”。

### 8. 历史 Vault 债务不能代表当前流程失败

旧入库结果可能保留已废弃格式，例如 `[!quote] 来源`、`原文摘录：` 或旧单文件个股逻辑。自测应对当前回归样本严格，对历史债务给出警告而不是直接判定当前工作流失败。

当前已知历史债：

- `obsidian-vault/行业知识库/航天军工电子.md`
- `obsidian-vault/个股逻辑/600879-航天电子.md`

处理原则：后续如果重新入库该标的，应按当前 archive 工作流覆盖/迁移，清理旧来源包装。

## 七、开发约定

### 1. Codex-only 优先

本项目已经切到 Codex 本地使用。新增功能优先考虑：

- 本地 Python
- MCP 工具
- Obsidian Markdown
- Codex agent 判断

不要重新引入外部 LLM API 作为必要路径。

### 2. 不破坏用户维护区

所有 vault 文件都应保留：

```text
<!-- /ai-content -->
```

该标记之后是用户维护内容。脚本只能改标记之前的 AI 管理区。

### 3. 不把 Obsidian 做成数据库垃圾场

进入 Obsidian 正文的内容必须满足至少一项：

- 能解释行业或个股长期逻辑。
- 能改变投资判断。
- 能作为后续跟踪指标。
- 能复用于同一行业其他标的。

不进入正文的内容：

- QA 全文
- YAML
- 工具调用记录
- 流程统计
- 免责声明
- 未改变逻辑的例行数据更新过程

### 4. 入库不是压缩研报

初次覆盖基线可以完整搬运关键逻辑，但后续跟踪不是“摘要压缩”。后续跟踪的目标是判断是否改变已有知识：

- 改变：更新相关逻辑。
- 不改变：记录本次结论和原文位置。

## 八、日报工作流

日报功能使用独立链路：

```text
/analyze-daily
    → 新闻/公告/政策 MCP 或 web 采集证据
    → reports/daily/YYYY-MM-DD/evidence.json
    → tools/daily_score.py
    → tools/daily_report.py
    → tools/daily_to_vault.py
    → obsidian-vault/日报/YYYY-MM-DD.md
```

日报时间窗口固定为分析指令发出当天及上一天，即上一日 00:00:00 至当天 23:59:59。日报是新闻整理和影响判断，不是行情日报；默认不调用行情、资金、估值或技术指标 MCP。日报默认不改写个股四面逻辑，重大事项只写入跟踪面板，后续由周度、季度或专题研究决定是否更新四面逻辑。
当 `scoring.json` 中的证据包含 `follow_up` 和 `related_stocks` 时，`tools/daily_to_vault.py` 会自动追加到对应单股跟踪面板。

新闻类 MCP 接入优先级：

- `official-announcement`：已配置接入，公司公告、交易所、监管。
- `macro-policy`：已配置接入，央行、统计局、财政部、发改委等。
- `market-news`：已配置接入，财联社、证券时报、上证报、中证报等。
- `industry-data`：已配置接入，行业政策、产业链事件、价格、库存、订单、供需新闻。

当前 MCP 接入检查：

- 已接入：`finance-data`、`tech-analysis`、`official-announcement`、`macro-policy`、`market-news`、`industry-data`。
- 已禁用：`akshare`。
- 注意：新增 MCP 写入 `.mcp.json` 后，当前 Codex 会话可能需要重启或重载 MCP 才会暴露对应工具。
- 日报默认不得使用 `finance-data`、`tech-analysis` 生成行情或技术内容；这两个 MCP 保留给个股跟踪、技术面分析和专题研究。

联网搜索只作为新闻类 MCP 缺口补充或交叉验证，不得替代已接入的官方源和结构化新闻 MCP。

日报相关脚本：

- `tools/daily_window.py`：生成“上一日 00:00:00 到当天 23:59:59”的日报窗口。
- `tools/daily_mcp_check.py`：检查新闻 MCP 本地配置、命令路径和服务器脚本状态；当前会话是否暴露工具仍需 Codex 用工具发现确认。
- `tools/daily_collect.py`：接收 MCP / web / manual 证据并生成 `evidence.json`，同时写入 MCP 暴露状态、四轮采集覆盖、窗口外剔除记录。
- `tools/daily_score.py`：对证据做来源、时效、关联度、验证度、发布时间置信度基础评分。
- `tools/daily_report.py`：生成 `日报.md`，包含信息覆盖度、运行日志、待核实池。
- `tools/daily_to_vault.py`：入库 `obsidian-vault/日报/YYYY-MM-DD.md`。

### 日报工作流已知问题与处理原则

- `.mcp.json` 配置成功不等于当前 Codex 会话已经暴露对应 MCP 工具。日报开始前必须做两层检查：本地配置检查 + 工具发现检查。
- 若新闻 MCP 未暴露，可以使用联网搜索兜底，但必须在 `--exposure-status` 和 `--exposure-note` 中留痕，并在日报“信息覆盖度与运行日志”中显示。
- 个股日报应优先扫描已覆盖股票和组合名单的公告，而不是泛泛搜索热门个股新闻。
- 只有单一来源、发布时间不精确、或找不到原始公告的信息，应进入“待核实信息”，不得写成确定性结论。
- 日报只触发知识库候选更新；除跟踪面板外，不直接改写个股四面逻辑或行业知识库正文。

### 日报开发经验沉淀

1. **MCP 分两层可用**

`.mcp.json` 和 server.py 正常，只能说明“本地配置可用”；Codex 当前会话是否暴露工具，需要在执行日报前用工具发现确认。新增 MCP 后，如果当前会话未暴露，不要反复怀疑代码，应提示重启/重载 MCP，或按 web fallback 执行。

2. **fallback 必须显式留痕**

日报允许在新闻 MCP 不可用时使用联网权威来源兜底，但不能静默降级。必须写入：

```bash
--exposure-status not_exposed
--exposure-note "当前 Codex 会话未暴露新闻 MCP，本次使用联网权威来源兜底。"
```

日报正文需要展示“信息覆盖度与运行日志”，让用户知道本次信息来源质量。

3. **时间精度不能伪装**

新闻只确认到日期时，不要强行写具体时分秒。使用：

```json
{
  "published_at_confidence": "low",
  "time_note": "来源只确认发布日期，未确认具体时间"
}
```

4. **待核实池是产品能力，不是失败**

找不到原始公告、只有单一媒体来源、时间不确定、或可信度不足的信息，应进入“待核实信息”。这样既不丢线索，也不污染正式结论。

5. **日报评分要服务投研判断**

评分不只是新闻热度排序，应综合：

- 来源等级
- 时效
- 时间置信度
- 多源验证
- 与已覆盖行业/个股的相关性
- 是否改变投资假设或触发跟踪任务

6. **日报不直接改知识库正文**

日报的角色是雷达和触发器。除跟踪面板外，原则上不直接改个股四面或行业知识库正文；重大事件先形成候选更新，再由周度、季度或专题研究决定是否改写长期逻辑。

## 九、后续可迭代方向

### 1. 更智能的逻辑合并

当前重大变化会追加到目标章节。未来可以支持：

- 查找同名子章节并替换局部段落。
- 给时效性数据自动标注日期。
- 对旧数据打“截至 YYYY-MM-DD”标签。

### 2. 投资决策日志优化

当前日志结论较模板化。未来可以由 Codex 在计划中填写更具体的 `decision_summary`，例如：

```json
{
  "decision_summary": "2026Q1 毛利率保持稳定，AI PCB 订单未见下修，核心逻辑不变，维持核心持仓但不追高。"
}
```

### 3. 跟踪面板自动维护

后续可增加：

- 从计划中提取跟踪指标。
- 自动更新指标状态。
- 指标触发阈值后提示需要深度复核。

### 4. 行业知识库去重

同一行业可能产生多个近似文件名，例如 `PCB.md`、`印制电路板（PCB）.md`、`AI服务器PCB.md`。未来需要：

- 行业别名表。
- 行业文件合并工具。
- 标的到行业的统一映射。

### 5. 文档统一

`AGENTS.md`、`README.md`、使用手册和 Skill 文档需要定期统一口径。每次改工作流都要检查是否出现旧描述。

## 十、当前回归基准

当前以沪电股份作为入库回归样本：

```text
reports/stocks/002463-沪电股份
reports/archive-plan-002463.json
obsidian-vault/个股逻辑/002463-沪电股份/
obsidian-vault/跟踪面板/002463-沪电股份.md
obsidian-vault/行业知识库/AI服务器PCB.md
```

已验证的预期状态：

- 四面文件无 `原文摘录：`。
- 四面文件无 `[!quote] 来源`。
- 无 `???` 乱码。
- 后续季度跟踪进入 `投资决策.md` 的“跟踪记录”。
- `原始报告/` 中保留 reports 下的 Markdown 原文副本。
- 跟踪面板在 `--overwrite` 后仍保留。
